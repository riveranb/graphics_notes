Vertex Shader：你會改的第一種程式碼

Application Vertex Shader Rasterization Fragment Shader Output Merging
這堂課結束後，你會：
  • 知道 Vertex Shader 在管線的哪個位置、負責什麼
  • 看懂一支 shader 的資料流：input 從哪來、output 往哪去
  • 分得清 attribute / uniform / varying 各自怎麼運作
  • 讀得懂 GLSL 常見型別與 swizzling（.xyz / .rgba）
  • 能自己動手改一行，讓表面做出 sin wave 波動

它在管線的哪、負責什麼

回到管線那條帶子：Vertex Shader 是第一個「可程式化」階段—— 前面的 Application 在 CPU 準備好資料，接著 GPU 就開始跑你寫的 vertex shader。

它的處理單位是頂點：Mesh 有幾個頂點，這支程式就跑幾次，每次處理一個。它唯一的硬性任務只有一件事：

算出這個頂點最後落在畫面的哪裡——也就是輸出 gl_Position（頂點在 Clip Space 的位置）。 這正是第 3 課那條「Object → World → Camera → Clip」的轉換，真正發生的地方。

先看它長怎樣

這是一支最基本的 Vertex Shader（GLSL）。別怕，等一下一行行拆：

attribute vec3 position;          // 輸入：這個頂點的位置（來自 Mesh）
attribute vec2 texCoord;          // 輸入：這個頂點的 UV
uniform   mat4 modelViewMatrix;   // 輸入：全域，Object→Camera 的矩陣
uniform   mat4 projectionMatrix;  // 輸入：全域，投影矩陣
varying   vec2 vTexCoord;         // 輸出：要傳給 Fragment Shader 的資料

void main() {
    gl_Position = projectionMatrix * modelViewMatrix * vec4(position, 1.0);
    vTexCoord = texCoord;
}

資料流：input 進來，output 出去

整支 shader 就是一台「加工機」：左邊餵資料進來，中間 main() 加工，右邊吐結果出去。

三個修飾字，看關鍵字就懂

那些寫在型別前面的字（attribute / uniform / varying）不是裝飾，它們指定了這個資料「從哪來、往哪去」：

  • attribute — 逐頂點輸入。 每跑一次換一個頂點的值（位置、UV、法線……就是第 2 課的頂點屬性）。
  • uniform — 全域輸入。 整個 draw call 裡所有頂點看到的都一樣，由 CPU 端設定一次（那兩個矩陣、時間、光源方向都是）。
  • varying — 輸出。 你算好、要交給 Fragment Shader 的資料。關鍵一點：它不是直接送達—— 會先在下一階段（Rasterization）被內插，三角形內每個像素拿到的是「按位置混合過」的值。這一段第 5 課會細講。
一句話記住三者：attribute 每個頂點不同、uniform 大家一樣、varying 往下傳（會被內插）。

GLSL 常見型別（讀 shader 的識字）

型別是什麼
float一個小數。
vec2 / vec3 / vec42 / 3 / 4 個 float 打包在一起。位置、顏色、UV 都是這種。
mat44×4 矩陣，就是那些變換矩陣。
sampler2D一張貼圖（第 6、7 課會用到）。

Swizzling：拆分量、重新組

一個 vec4 裡的分量，可以用兩套名字叫——同一塊資料，只是別名：

  • .x .y .z .w — 當座標看時用
  • .r .g .b .a — 當顏色看時用

而且可以任意挑、重排、甚至重複同一個分量，這動作叫 swizzling：

vec4 c = vec4(0.1, 0.2, 0.3, 1.0);

c.xy       // → vec2(0.1, 0.2)      取前兩個
c.rgb      // → vec3(0.1, 0.2, 0.3) 只取顏色三分量
c.bgr      // → vec3(0.3, 0.2, 0.1) 調換順序
c.xxx      // → vec3(0.1, 0.1, 0.1) 重複同一個（灰階填滿 RGB 常這樣）

vec4(position.xyz, 1.0)   // 取 xyz 再補一個 w=1.0 —— 每支 shader 都看得到

那行矩陣乘法：順序不能反

再看核心那一行：

gl_Position = projectionMatrix * modelViewMatrix * vec4(position, 1.0);

矩陣乘法從右往左作用：先 modelViewMatrix 把頂點帶到 Camera Space，再 projectionMatrix 投影到 Clip Space。 矩陣乘法不能交換——順序一調，頂點就被算到錯的地方，畫面整個壞掉。這也是「東西沒出現、位置歪掉」時第一個要檢查的地方。

動手改一行：讓表面做出 sin wave

Vertex Shader 既然決定每個頂點的位置，那我們就能在送去投影之前，先動它的座標。 這就是水面、旗子飄動這類效果的做法——一支 shader，改幾行而已： 

uniform float uTime;                    // 一直變大的時間（CPU 每幀傳進來）

void main() {
    vec3 pos = position;
    pos.y += sin(pos.x + uTime);        // 依 x 起伏、隨時間前進 → 波
    gl_Position = projectionMatrix * modelViewMatrix * vec4(pos, 1.0);
}

波動要成立，這一行需要三個零件同時到位：sin()（才有波形）、pos.x（波形隨空間變化）、uTime（波才會往前跑）。缺一個就不是「會動的波」。

對應到 Cocos：上面用 uniform float uTime 是通用寫法（你得自己宣告、自己每幀傳）。 在 Cocos Creator 你不用自己宣告——引擎內建全域 uniform cc_time（vec4）， 直接用 cc_time.x 就是「啟動至今的秒數」。所以在 Cocos 裡那行會寫成 pos.y += sin(pos.x + cc_time.x);。

它跑很多次，而且很平行

模型有幾個頂點，這支程式就跑幾次，而且 GPU 是大量同時平行算。 所以 vertex shader 要簡單快——每個頂點都要付一次成本。

它做不到什麼

劃清邊界，免得誤會它萬能：

  • 不能決定最終顏色——那是 Fragment Shader 的工作（第 6 課）。
  • 不能增減頂點數量——它只能搬動既有頂點，不能無中生有或刪掉（那要別的 shader）。
