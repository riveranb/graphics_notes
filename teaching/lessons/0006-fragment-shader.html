Fragment Shader：決定「顏色」的那支程式

Application Vertex Shader Rasterization Fragment Shader Output Merging
這堂課結束後，你會：
  • 知道 Fragment Shader = 每個片段/像素跑一次的小程式
  • 知道它的工作：算出這個像素的最終顏色
  • 建立除錯直覺：顏色不對，先來這裡找

第二種你會改的程式碼

Vertex Shader 決定「頂點在哪」，Fragment Shader 決定「每個像素是什麼顏色」。 畫面太亮太暗、顏色怪、貼圖沒出來 —— 十之八九改這支。一支最簡單的長這樣： 

varying vec2 vTexCoord;            // 從 Vertex Shader 內插來的 UV
uniform sampler2D diffuseMap;     // 一張綁定的貼圖

void main() {
    vec4 texColor = texture2D(diffuseMap, vTexCoord);  // 用 UV 去貼圖取色
    gl_FragColor = texColor;                            // 輸出：最終顏色
}

輸入輸出

  • varying/in — 輸入：從 Vertex Shader 傳來、被光柵化內插過的資料（UV、法線…）
  • uniform — 輸入：全域值，如貼圖 sampler2D、光照參數
  • gl_FragColor — 輸出：這個片段的最終顏色 (r,g,b,a)

上面那行 texture2D(diffuseMap, vTexCoord) 就是「拿 UV 去貼圖上取顏色」—— 正是下一課 Texture 的主題。

兩支 shader 的分工，一句話：Vertex Shader 算位置，Fragment Shader 算顏色。

它會跑「非常」多次

畫面上每個被蓋到的像素都要跑一次 fragment shader —— 通常比頂點多幾個數量級。 所以這裡的每一行都很貴，效能問題常出在這支。
