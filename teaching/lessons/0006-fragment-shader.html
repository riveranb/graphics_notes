Fragment Shader：決定「顏色」的那支程式

Application Vertex Shader Rasterization Fragment Shader Output Merging
這堂課結束後，你會：
  • 知道 Fragment Shader 在管線的哪個位置、每次處理什麼
  • 看懂它的資料流：內插過的 varying 進來、gl_FragColor 出去
  • 會三個實戰改法：染色（tint）、UV 除錯顯示、用時間做呼吸閃爍
  • 建立除錯直覺：顏色不對，先來這裡查什麼

它在管線的哪、負責什麼

接著第 5 課：Rasterization 把三角形填成一格格Fragment（片段）， 每個片段帶著內插好的資料（UV、顏色、法線…）。 Fragment Shader 就是每個片段跑一次的小程式， 它唯一的硬性任務：算出這個片段的最終顏色——輸出 gl_FragColor。

兩支 shader 的分工，一句話：Vertex Shader 算位置，Fragment Shader 算顏色。 畫面太亮太暗、顏色怪、貼圖沒出來——十之八九改這支。

先看它長怎樣

一支最常見的 Fragment Shader（GLSL），一行行拆：

varying vec2 vTexCoord;           // 輸入：Vertex Shader 傳來、內插過的 UV
uniform sampler2D diffuseMap;     // 輸入：全域，一張綁定的貼圖

void main() {
    vec4 texColor = texture2D(diffuseMap, vTexCoord);  // 拿 UV 去貼圖取色
    gl_FragColor = texColor;                            // 輸出：最終顏色 (r,g,b,a)
}

texture2D(貼圖, UV) 的細節（取樣、filtering）是下一課 Texture 的主題，這裡先當「去貼圖上拿顏色」就好。

資料流：跟 Vertex Shader 一樣的讀法

  • varying — 輸入：Vertex Shader 傳出、 在 Rasterization 被內插過的資料。到手的不是某個頂點的原值，是「按位置混合過」的值（第 5 課的 lerp）。
  • uniform — 輸入：全域值。貼圖 sampler2D、時間、光照參數都走這。
  • gl_FragCoord — 唯讀（read-only）內建輸入：這個片段在螢幕上的像素座標 (x, y)。你只能讀它、不能寫。
  • gl_FragColor — 輸出：這個片段的最終顏色 (r,g,b,a)。 GPU 的顏色預設有效值域是 [0, 1]：大於 1 當成 1、小於 0 當成 0（會被夾住 / clamp）。 所以 ③ 的 sin() 負值會被夾成全黑，才需要 ×0.5 + 0.5 先搬進 [0, 1]。

動手改：三個實戰小改法

Fragment Shader 是你最常動手的地方。三個真實工作裡天天見的改法，每個下面都附一塊即時預覽：

下面的預覽是純 WebGL 在你瀏覽器裡即時執行的（不是圖片）。兩個小代換：底色用程序棋盤代替貼圖、 cc_time.x 由預覽器幫你填（在 Cocos 是引擎自動填）——但你要學的那一行邏輯完全一樣。

① 染色（tint）：乘一個顏色

uniform vec4 uTintColor;          // 自己宣告的全域值，CPU 端設定

void main() {
    vec4 texColor = texture2D(diffuseMap, vTexCoord);
    gl_FragColor = texColor * uTintColor;   // 逐分量相乘：r×r, g×g, b×b, a×a
}

vec4 × vec4 是逐分量相乘（第 4 課的 vec4，每個位置各乘各的）。 tint 設 (1,1,1,1) 等於不變；設 (1,0,0,1) 就只剩紅色分量——角色受傷變紅、UI 反灰都是這招。

② UV 除錯顯示：把資料當顏色印出來

void main() {
    gl_FragColor = vec4(vTexCoord, 0.0, 1.0);   // u→紅、v→綠
}

貼圖貼歪的時候，工程師會暫時把 UV 直接當顏色輸出：u 進紅色、v 進綠色。 UV (0,0) 的角落變黑、(1,0) 變紅、(0,1) 變綠、(1,1) 變黃—— 畫面會是一片平滑漸層，那個漸層就是第 5 課的內插正在發生。 漸層不對（斷裂、方向反了），就知道是 UV 資料壞了,不是貼圖壞了。

③ 呼吸閃爍：用時間動起來

void main() {
    vec4 texColor = texture2D(diffuseMap, vTexCoord);
    float breath = sin(cc_time.x) * 0.5 + 0.5;   // -1~1 重新映射到 0~1
    gl_FragColor = texColor * breath;
}

sin() 的輸出是 -1～1，但顏色只吃 0～1（負的會被夾成 0，等於有一半時間全黑）。 所以先 ×0.5 壓成 -0.5～0.5、再 +0.5 抬到 0～1—— 這個「×0.5 + 0.5 重新映射」在 shader 裡到處都是，記下來。

對應到 Cocos：cc_time 是引擎內建全域 uniform（vec4），不用自己宣告， cc_time.x = 啟動至今的秒數——第 4 課 sin wave 用過的同一個。上面預覽的 cc_time.x 就是預覽器幫你填的。

它會跑「非常」多次

一個 1920×1080 的畫面約有 200 萬個像素；而一個普通模型可能只有幾千個頂點。 也就是說 fragment shader 的執行次數，動輒是 vertex shader 的幾百到幾千倍—— 同一行浪費的程式碼，在這裡的代價放大幾個數量級。效能問題常出在這支，就是這個原因。

還有一個硬體細節：GPU 不是一次跑一個 pixel，而是以 2×2 一組（一個 quad）為最小單位一起跑。 後果是——就算三角形邊緣只蓋到 1 個 pixel，那整組 2×2 還是會跑滿四份（另外三份算完就丟）。 所以又多又小的三角形（過度細分）在邊緣會有大量這種浪費，比想像中更吃效能。

它做不到什麼

  • 不能動位置——片段的螢幕位置在光柵化時就定了，這裡只能決定顏色（要動位置回第 4 課）。
  • 不能直接跟畫面上已有的顏色混合——半透明疊色是下一階段 Output Merging 的事（第 9 課 Blending）。這支只負責交出「我這個片段的顏色」。
