# Notes

## Workspace 決策（2026-07-01）
- 用途：不是自學，是**做教材給別人**——受眾是團隊裡零圖學背景的工程師（不分資淺資深）。
- 交付形式：純自學，學生自己依序看 lessons，不搭配排定的上課時間 → lessons 必須完全自足，不能留「上課再講」的坑。
- Workspace 位置：使用者選擇放在這個筆記 repo 底下的 `teaching/` 子目錄（而非獨立 repo），筆記與教材共用同一個 git history。
- 教材範圍優先序：讀懂/能修改現有渲染程式碼 > 獨立抓效能問題。效能優化系列（遊戲優化、CocosCreator優化）目前是 out of scope，見 MISSION.md。

## 素材特性
- 來源筆記本身就是設計給「已審閱過」的品質（`.agent/skills/graphics_article_reviewer`），可以放心當 Primary Knowledge，不用另外驗證。
- 筆記大量使用 `images/*.png|svg|gif`，且圖選得很到位——lesson 直接 hotlink 這些既有圖片（相對路徑 `../../images/...`），不要重新畫圖或複製一份。
- 筆記的「參考延伸閱讀」段落本身就是高信任外部來源，做 lesson 時直接搬進 RESOURCES.md，不用重新找。

## 教學設計取捨
- 學生零圖學背景 → 管線地圖先行，數學延後且降級為可選參考（見下「核心教學原則」）。

## 核心教學原則（使用者明確要求，最高優先）
- **輕量、體驗導向**：寧可「輕而多課」，不要「重而少課」。每課單一概念，草稿覺得重就拆——就算 count 因此微升也沒關係。
- **數學絕不當關卡**：數學只給直覺 + 程式碼裡真的會看到的那一行（如 `gl_Position = MVP * pos`）；矩陣推導／證明一律連出去到 遊戲數學 筆記或放 `reference/`，不內嵌。繁雜理論與數學造成壓力會有反效果。
- **每課開頭用看得見的結果或「哪裡會壞」**，不要用定義開場。
- **一個互動勝過三段文字**：每課一個開關／拖曳／測驗。
- 詳細課程大綱見 [SYLLABUS.md](SYLLABUS.md)。
